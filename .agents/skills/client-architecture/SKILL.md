# Архитектура Frontend (Client)

**ОБЛАСТЬ ДЕЙСТВИЯ:** Эти правила применяются ТОЛЬКО к файлам внутри директории `apps/client/`. Если ты редактируешь сервер или парсеры, ИГНОРИРУЙ этот файл.

Ты работаешь в проекте со строгой слоевой архитектурой (Layered Architecture / Vertical Slices).
При создании или изменении файлов ты ОБЯЗАТЕЛЬНО должен соблюдать однонаправленный поток зависимостей.

## ГЛАВНОЕ ПРАВИЛО ИМПОРТОВ

Слой может импортировать ТОЛЬКО из слоев с меньшим числовым префиксом. Импорты "вверх" ЗАПРЕЩЕНЫ. Фичи не могут импортировать другие фичи.

## Слои в директории `components/`

- **`01.kit` (UI Kit)**: Атомарные компоненты (`kit-btn`, `kit-input`).
  - _Правило для ИИ:_ НИКАКОГО стейта (Pinia), НИКАКИХ API-запросов. Только props и emits. Не импортируют ничего из других слоев.
- **`02.shared` (Виджеты)**: Общие блоки (`app-header`, `theme-manager`).
  - _Правило для ИИ:_ Могут использовать только компоненты из `01.kit`.
- **`03.domain` (Бизнес-сущности)**: Сложная логика (например, геолокация).
  - _Правило для ИИ:_ Могут импортировать `01.kit` и `02.shared`.
- **`04.features` (Фичи)**: Вертикальные срезы (UI + State + Models). Пример: `trip-info/trip-comments`.
  - _Правило для ИИ:_ Каждая фича ИЗОЛИРОВАНА. Если ты создаешь фичу, ее Pinia-store должен лежать внутри папки фичи (например, `04.features/.../store/`). Запрещено напрямую вызывать одну фичу из другой.
- **`05.modules` (Страницы/Блоки)**: Композиционный слой (`trips-hub`, `trip-info`).
  - _Правило для ИИ:_ Здесь мы собираем фичи и виджеты вместе. Могут импортировать слои `01`–`04`.
- **`06.layouts`**: Шаблоны страниц (`default`, `empty`).

## Внешние директории (Важно)

- **`pages/`**: Файлы здесь ТОЛЬКО для роутинга. Они должны быть максимально тонкими и просто импортировать страницу из слоя `05.modules`. Не пиши здесь бизнес-логику!
- **`shared/services/database`**: Вся работа с API должна идти только через этот слой (через клиенты `trpc`, `sql`, `mock`). Не используй `useFetch` напрямую в компонентах фич, обращайся к сервисам.
- **`shared/store`**: Только глобальные Pinia-сторы (Auth, Theme). Сторы фич лежат в `04.features`.

## Инструкция по самопроверке для ИИ

Перед тем как сгенерировать код и сохранить файл, мысленно проверь:

1. В какую папку я кладу этот файл?
2. Не нарушаю ли я правило префиксов `01-06` в секции `<script setup>` при импортах?
3. Изолирована ли бизнес-логика (в `04.features` или `05.modules`) от тупых UI-компонентов (`01.kit`)?
4. **НЕ ПЛОДИ ДУБЛИ:** Перед тем как создавать новый UI-компонент, ОБЯЗАТЕЛЬНО прочитай содержимое папки `components/01.kit` и `components/02.shared`. Если там уже есть подходящий компонент (например, `kit-btn` или `kit-input`), импортируй его, а не пиши HTML-теги с нуля.

## Документация (MCP)
При работе с этим клиентским приложением, если ты не уверен в синтаксисе Vue 3, Pinia или VueUse, ОБЯЗАТЕЛЬНО используй инструменты из MCP-сервера `vue-docs` для поиска актуальной информации.

# Технический стек и Стилизация

1. **Стили:** Мы используем [SCSS / Vue style scoped]. НИКОГДА не пиши инлайн-стили (`style="..."`).
2. **Иконки:** Для иконок мы используем `import { Icon } from '@iconify/vue'`. Запрещено вставлять сырые SVG прямо в шаблон, используй компонент `<Icon icon="..." />`.
3. **Язык:** Всегда используй TypeScript (`<script setup lang="ts">`).
